# 10 萬人同時線上優化方向分析

> 分析日期：2026-03-07
> 基準版本：main branch（commit 046aa5b）
> 架構：Node.js + Express + PostgreSQL（單機）

---

## 目前架構瓶頸總覽

| 層級 | 問題 | 嚴重程度 |
|------|------|----------|
| 進程 | 單一 Node.js 進程，無 Cluster | 嚴重 |
| 資料庫 | Pool 無設定、無讀寫分離、無快取 | 嚴重 |
| 中介層 | IP 白名單阻擋所有外部用戶 | 嚴重 |
| 認證 | bcrypt 每次請求阻塞 CPU | 高 |
| 日誌 | 每次請求執行 jwt.verify + 磁碟 I/O | 中 |
| 限流 | 100 次/15 分鐘全域限制 | 中 |
| 部署 | 無負載均衡、無水平擴展機制 | 嚴重 |

## 一、進程層（Process）

### 問題
`server.js` 單一進程，Node.js 是單執行緒，只能用到 1 個 CPU 核心。
10 萬並發 → 事件循環塞滿 → 請求逾時。

### 優化方向

**1. Node.js Cluster**
```js
// 利用所有 CPU 核心
const cluster = require('cluster');
const os = require('os');
if (cluster.isPrimary) {
  for (let i = 0; i < os.cpus().length; i++) cluster.fork();
} else {
  require('./server');
}
```

**2. PM2 多進程管理（生產推薦）**
```bash
pm2 start server.js -i max   # 自動對應 CPU 核心數
pm2 start server.js -i 4     # 固定 4 個 worker
```

**3. 水平擴展（多台機器）**
- 前置 Nginx / HAProxy 做 Load Balancer
- 無狀態設計（JWT 已做到，符合條件）
- Docker + Kubernetes 水平 Pod 擴展

## 二、資料庫層（PostgreSQL）

### 問題
`db/index.js` 中 `new Pool()` 沒有設定任何參數，預設最大連線 **10**。
10 萬並發 → 連線等待佇列爆滿 → 全面超時。

### 優化方向

**1. 調整 Pool 設定**
```js
const pool = new Pool({
  max: 50,              // 每個 Node 進程最大連線數
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
});
```
> 注意：PostgreSQL 服務端 `max_connections` 也要對應調整（預設 100）

**2. PgBouncer 連線池代理**
- 在 Node.js 與 PostgreSQL 之間加一層 PgBouncer
- Transaction 模式：可支撐數千並發，實際 DB 連線維持在低數量
- 對應 10 萬用戶：`pool_size = 100~200` 即可撐住

**3. 讀寫分離**
- Primary（寫）處理 INSERT / UPDATE / DELETE
- Replica（讀）處理 SELECT 查詢（行事曆查詢、用戶資訊）
- `getSchedulesByUserId`、`findByIdAndUserId` 全部導向 Replica

**4. 資料庫索引**
```sql
-- 行事曆查詢頻繁用到的條件
CREATE INDEX idx_schedules_user_id_start_time ON schedules(user_id, start_time);
CREATE INDEX idx_schedules_user_id_end_time ON schedules(user_id, end_time);

-- 用戶查詢
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_account ON users(account);

-- 參加者查詢
CREATE INDEX idx_participants_user_schedule ON participants(user_id, schedule_id);
```

**5. Redis 快取層**
- 快取用戶基本資訊（`/api/user/info`）：TTL 5 分鐘
- 快取行事曆列表：TTL 30 秒（資料更新時 invalidate）
- JWT 黑名單（登出 token 失效）也可用 Redis 處理

## 三、中介層（Middleware）

### 問題 A：IP 白名單阻擋所有外部用戶
`ipWhitelist.js` 只允許 `.env` 中設定的 3 個 IP，**所有真實用戶都會拿到 403**。

**優化方向**
- 若白名單只用於管理/測試端點，應移至對應路由，而不是全域套用
- 若是環境隔離設計，生產環境應改為黑名單模式（封鎖惡意 IP），或交由 WAF / Nginx 處理

### 問題 B：Logger 每次請求都執行 jwt.verify
```js
// 每個請求都做一次 JWT 解密（即使後續 auth middleware 也會做）
const decoded = jwt.verify(authHeader.split(' ')[1], process.env.SECRET);
```
這導致每個請求有額外一次 CPU 密集操作。

**優化方向**
- Logger 改用 `jwt.decode`（不驗證簽章，純解析），只用於記錄 user_id，驗證留給 auth middleware
- 或從 `req.user`（已驗證後附加）讀取，避免重複解析

### 問題 C：同步磁碟 I/O 日誌
`fs.appendFile` 每次請求寫一次磁碟。10 萬 QPS 下磁碟 I/O 成為瓶頸。

**優化方向**
- 使用 **winston** 或 **pino**（非同步 stream、批次寫入）
- pino 效能遠優於 console.log + fs.appendFile
- 生產環境考慮將日誌導向 stdout，由 PM2 / k8s log aggregator 收集

## 四、認證層（Auth）

### 問題：bcrypt 阻塞事件循環
`bcrypt.hash(password, 10)` 與 `bcrypt.compare()` 是 CPU-bound 操作。
雖然 `bcrypt` npm 套件使用 C++ binding + libuv thread pool，但預設 thread pool 只有 4 個。
大量並發登入 / 註冊時，其他 async 操作也會被拖慢。

**優化方向**
- 將 bcrypt 工作移至獨立 Worker Thread，不佔用 main thread pool
- 調高 `UV_THREADPOOL_SIZE`（環境變數，最高 128）
- 考慮使用 **argon2**（更現代，同樣安全，效能略優於 bcrypt）

### 問題：Token 無法主動失效
目前 JWT 無黑名單機制，登出後 token 在 1 小時內仍有效。

**優化方向**
- Redis 儲存登出 token 的 jti（JWT ID），auth middleware 每次驗證時查詢
- 或縮短 token 有效期（15 分鐘） + Refresh Token 機制

## 五、限流策略

### 問題：全域 100 次/15 分鐘過於嚴格
10 萬用戶，每人每 15 分鐘只有 100 次請求，正常使用下很快就會觸發。

**優化方向**
- 按用途分層限流：
  ```
  /api/auth/login     → 每 IP 每分鐘 5 次（防暴力破解）
  /api/auth/register  → 每 IP 每小時 10 次
  /api/schedule/*     → 每用戶每分鐘 60 次（正常操作）
  全域               → 每 IP 每分鐘 200 次（防 DDoS）
  ```
- 使用 Redis 作為 Rate Limit 的儲存後端（多進程共享計數）
  ```js
  const RedisStore = require('rate-limit-redis');
  // express-rate-limit 本身支援 store 選項
  ```

## 六、排程衝突檢查（Race Condition）

### 問題：非原子操作導致 Race Condition
`createSchedule` 的流程：
1. `SELECT` 查詢是否有時段衝突
2. 若無 → `INSERT` 新行程

步驟 1 和步驟 2 之間沒有鎖，10 萬並發下可能兩個請求同時通過步驟 1，造成重複建立。

**優化方向**
- 使用 PostgreSQL `SERIALIZABLE` 隔離等級 + Transaction
- 或在資料庫層加 Unique Constraint（時段不重疊的 exclusion constraint）
  ```sql
  -- 使用 pg_trgm 或 tsrange 的 exclusion constraint
  ALTER TABLE schedules
  ADD CONSTRAINT no_overlap EXCLUDE USING gist (
    user_id WITH =,
    tsrange(start_time, end_time) WITH &&
  );
  ```

## 七、架構總覽（目標狀態）

```
用戶 (10萬)
    ↓
[CDN / DDoS 防護（Cloudflare）]
    ↓
[Nginx Load Balancer]
    ↓
[Node.js 進程 x N（PM2 / k8s Pods）]
    ↓         ↘
[Redis]    [PgBouncer]
快取/限流       ↓
         [PostgreSQL Primary]
              ↓
         [PostgreSQL Replica(s)]
              ↓
         [備份 / WAL 歸檔]
```

## 優化優先序

| 優先 | 項目 | 預期效益 |
|------|------|----------|
| P0 | IP 白名單改為路由層或移除 | 解除全部用戶被封鎖 |
| P0 | PM2 多進程 / Cluster | 線性擴展 CPU 利用率 |
| P0 | PgBouncer + Pool 調整 | 解除 DB 連線瓶頸 |
| P1 | Redis 快取 + Rate Limit store | 降低 DB 壓力、多進程共享限流 |
| P1 | DB 索引建立 | 查詢從全表掃描變索引掃描 |
| P1 | Logger 改用 pino | 降低 I/O 壓力 |
| P2 | 讀寫分離（Replica） | 進一步分散讀取壓力 |
| P2 | Schedule 衝突 DB 層約束 | 解決 Race Condition |
| P2 | JWT Refresh Token + Redis 黑名單 | 安全性提升 |
| P3 | Kubernetes 水平自動擴展（HPA） | 動態應對流量峰值 |
